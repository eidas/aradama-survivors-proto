# 自律開発ループ計画(ゴールと判定基準)

> コーディネーター+Coder/Reviewer サブエージェントによる長時間自律開発の運用規約。
> バックログ本体は [BACKLOG.md](../BACKLOG.md)、エージェント向け要約は [CLAUDE.md](../CLAUDE.md)。

## 1. 北極星ゴール

**「刀使ノ巫女ファンとサバイバーライク好きが、ブラウザで15分遊んで『もう1回』と思える完成度のプロトタイプ」**

定量分解した維持目標:

| ID | ゴール | 測定方法 |
|---|---|---|
| G1 安定性 | `npm run verify`(型+テスト+ビルド+実機スモーク)が常時グリーン | verify の exit code |
| G2 性能 | 敵300体ストレス時にスモーク環境の基準fpsから 20% 以内の低下 | verify のスモーク計測値 |
| G3 コンテンツ | BACKLOG の MUST 項目の消化 | BACKLOG の DONE 数 |
| G4 健全性 | データ駆動の維持(数値は `src/data/` のみ)、レビュー blocking 指摘の再発ゼロ | Reviewer 判定履歴 |

## 2. 体制と責務

| 役割 | 担当 | 責務 | 持たない責務 |
|---|---|---|---|
| コーディネーター | メインセッション | タスク選定、エージェント起動、ゲート判定、BACKLOG更新、エスカレーション | コードの実装・詳細レビュー(トークン節約のため要約のみ保持) |
| Coder | サブエージェント | 1タスク=1コミット。CLAUDE.md読込→実装→`npm run verify`→コミット | タスクのスコープ変更、BACKLOG編集 |
| Reviewer | サブエージェント | コミットdiffのレビュー専任。APPROVE / REQUEST_CHANGES 判定 | コードの編集(読み取り専用) |

## 3. イテレーションの段階と判定基準(ゲート)

```
Gate 0 選定 → Coder実装 → Gate 1 自己検証 → Reviewer → Gate 2 レビュー → Gate 3 統合 → (3タスクごと) Gate 4 ループ健全性
```

### Gate 0: タスク選定(コーディネーター)
- BACKLOG の最上位優先度の TODO であること
- **1コミットに収まる粒度**であること(目安: 変更 300 行以内)
- 受け入れ条件が検証可能な形で書けること
- 満たさない場合 → タスクを分割して BACKLOG を書き換えてから着手

### Gate 1: 実装完了(Coder の完了条件)
- `npm run verify` が **PASS**(型チェック / 全テスト / ビルド / 実機スモーク)
- 新規ロジックには単体テストを追加(描画のみの変更は除く)
- バランス数値の変更は `src/data/` に限定
- 受け入れ条件を満たすことを Coder 自身がスモークまたはテストで確認済み
- コミット済み(メッセージは CLAUDE.md の規約)

### Gate 2: レビュー(Reviewer の判定基準)
- **APPROVE**: blocking 指摘なし → Gate 3 へ
- **REQUEST_CHANGES**: blocking 指摘あり → Coder に差し戻し(指摘リスト添付)
- blocking の定義: バグ、仕様(docs/03)との不整合、データ駆動違反、毎フレームのアロケーション追加、テスト欠落
- non-blocking(提案)は記録のみで差し戻さない
- **差し戻しは同一タスク最大2回**。超過したらコーディネーターがタスクを BLOCKED にして次へ

### Gate 3: 統合(コーディネーター)
- コミットがリモートに反映されている(auto-push hook の発火確認、未反映なら手動 push)
- BACKLOG のステータス更新(DONE / BLOCKED)
- 受け入れ条件に UI 変更が含まれる場合、スクリーンショット確認(コーディネーターが verify の成果物で確認)

### Gate 4: ループ健全性(3タスクごと、コーディネーター)
- `npm run verify` の所要時間 < 5分(超過したらスモークの整理をタスク化)
- テスト件数が減っていない
- 直近コミットに revert がない
- 満たさない場合 → 新規タスクを止めて健全化タスクを優先

## 4. 停止・エスカレーション基準(人間に返す条件)

| 状況 | 対応 |
|---|---|
| 同一タスクが2回差し戻し | BLOCKED にして次のタスクへ(人間への報告リストに積む) |
| verify が環境起因で2回連続失敗 | ループ停止、状況を報告 |
| 仕様の解釈・原作設定の判断・スコープ拡大が必要 | 実装せず質問として報告 |
| BACKLOG の TODO が枯渇 | 新規候補リストを提案して指示を待つ |
| 破壊的操作(履歴書き換え、削除、外部公開設定の変更)が必要 | 必ず人間の承認を待つ |

## 5. トークン・コンテキスト運用規約

- 各エージェントは **CLAUDE.md + 担当タスクに関係するファイルのみ**読む。docs/ の全読み禁止(必要節のみ)
- Coder への指示は「タスクID+受け入れ条件+制約」のみ。実装方針の長文指示はしない(CLAUDE.md に共通規約を集約)
- Reviewer には「コミットハッシュ+受け入れ条件」のみ渡し、diff は自分で `git show` させる
- コーディネーターはエージェントの**最終報告の要約のみ**を保持し、中間ログを引き取らない
- 検証は常に `npm run verify` の1コマンドに集約(エージェントごとの独自検証手順を作らない)
