## 1. 技術選定

| 項目 | 採用 | 理由 |
|---|---|---|
| エンジン | **Phaser 3** | 2D 特化で軽量。WebGL バッチ描画で「数百体の敵+大量のジェム」というサバイバーライク要件に十分。ライセンス費なし(MIT) |
| 言語 | **TypeScript** | データ駆動設計(キャラ・敵・強化の定義)を型で保証できる。リファクタ耐性 |
| ビルド | **Vite** | 高速な開発サーバ・HMR。静的サイトとしてそのままデプロイ可能 |
| 配布 | 静的ホスティング(GitHub Pages 等) | ファンメイド作品として URL 共有だけで遊べることを最優先 |
| テスト | **Vitest** | 描画に依存しないロジック(ダメージ計算、ウェーブ進行、レベルアップ抽選)を単体テスト |

**代替案として棄却したもの**
- *Godot 4*: エディタ込みで開発体験は良いが、Web エクスポートのバイナリサイズ・起動時間と、コードレビュー中心のリポジトリ運用との相性で今回は見送り。ネイティブ配布が必要になったら再検討。
- *Unity*: 2D プロトタイプには過剰。ライセンスと Web 出力の重さから除外。
- *自前 Canvas 実装*: 制御は最大だが、入力・オーディオ・シーン管理を再発明するコストが見合わない。

## 2. 性能目標

| 指標 | 目標値 |
|---|---|
| フレームレート | 60 fps(デスクトップブラウザ) |
| 同時アクティブ敵数 | 300 体 |
| 同時ノロジェム数 | 1,000 個(自動マージで抑制) |
| 同時エフェクト/ダメージ表示 | 200 |
| 初回ロード | 5 秒以内(アセット合計 ~10MB 目安) |

## 3. アーキテクチャ方針

### 3.1 全体構成

Phaser の Scene をトップレベルの状態機械とし、ゲームプレイ内は **軽量 ECS 風(Entity + System)** で構成する。フル ECS ライブラリは導入せず、「エンティティ = プールされたオブジェクト、システム = 更新順が固定された処理クラス」に留める。

```
Boot → Title → CharacterSelect → Game ⇄ LevelUp(オーバーレイ) → Result
```

### 3.2 ゲームプレイ内システム(毎フレーム、固定順で更新)

1. **InputSystem** — キーボード/パッド入力の正規化
2. **AbilitySystem** — 迅移・金剛身・八幡力・写しの状態遷移とゲージ管理
3. **PlayerSystem** — 移動・攻撃発生
4. **SpawnSystem** — ウェーブテーブルに従う敵生成(プールから取得)
5. **EnemyAISystem** — タイプ別ステートマシン(追尾/周回急降下/突進/等)
6. **MovementSystem** — 全エンティティの位置更新
7. **CollisionSystem** — 空間ハッシュによる円衝突判定
8. **CombatSystem** — ダメージ適用、荒魂の自動再生、撃破 → ノロ変換
9. **PickupSystem** — ノロ吸引・回収、経験値加算
10. **LevelUpSystem** — 閾値到達でオーバーレイ起動、強化抽選・適用
11. **HUD/VFXSystem** — 表示更新(ロジックから分離)

### 3.3 データ駆動

キャラ・能力・敵・ウェーブ・強化はすべて **TypeScript の定義データ(`src/data/`)** に外出しし、コードはそれを解釈するだけにする。バランス調整が定義ファイルの数値変更だけで完結することを設計原則とする。スキーマは 04 で定義。

### 3.4 パフォーマンス戦略

- **オブジェクトプール**: 敵・ジェム・エフェクト・ダメージ数字は全てプール。ラン中の new/GC を発生させない。
- **空間ハッシュ**: 衝突判定はセル分割(セルサイズは敵最大半径基準、04 で規定)で近傍のみ照合。物理エンジン(Matter 等)は使わない。
- **ジェムのマージ**: 密集したノロジェムは 1 個の高価値ジェムに自動統合し、上限 1,000 個を守る。
- **描画**: 単一テクスチャアトラス+WebGL バッチ。画面外エンティティは描画スキップ(ロジックは継続)。

## 4. 入力

| 操作 | キーボード/マウス | ゲームパッド |
|---|---|---|
| 移動 | WASD / 矢印 | 左スティック |
| 通常攻撃 | 自動(最寄りの敵方向) | 自動 |
| 八幡力(溜め) | 攻撃キー長押し(Space) | R2 長押し |
| 迅移 | Shift | L2 / A |
| 金剛身 | 右クリック / E 長押し | L1 長押し |

- 通常攻撃はサバイバーライク慣習に従い**オート**。能動操作は迅移・金剛身・八幡力の 3 つに絞り、操作負荷を抑える。

## 5. アセット計画

| 段階 | 内容 |
|---|---|
| フェーズ 1(プロト) | 図形+シルエットのプレースホルダ。荒魂は黒ベース+赤発光コアの手続き的表現で雰囲気を先行検証 |
| フェーズ 2 | 差し替え可能なスプライトアトラス(キャラ 3、敵 6、エフェクト)。BGM/SE はフリー素材から仮当て |

- 権利面: 原作の画像・音声素材は使用しない。全アセットはオリジナル制作またはライセンス上問題のないフリー素材に限定する。

## 6. リポジトリ構成(概要)

```
/
├── docs/          … 本設計ドキュメント群
├── src/
│   ├── scenes/    … Phaser Scene(Boot/Title/CharacterSelect/Game/Result)
│   ├── systems/   … §3.2 の各システム
│   ├── entities/  … Player / Enemy / Gem などのプール対象
│   ├── data/      … キャラ・敵・ウェーブ・強化の定義データ
│   ├── core/      … 空間ハッシュ、プール、乱数、数式ユーティリティ
│   └── ui/        … HUD、レベルアップ 3 択、リザルト
├── public/assets/ … 画像・音声
└── tests/         … Vitest(ロジック単体テスト)
```

## 7. セーブ・設定

- プロトタイプでは `localStorage` に「ベスト記録(生存時間・撃破数)」「音量等の設定」のみ保存。メタ進行は持たない(01 §8 のスコープに準拠)。

## 8. リスクと対応

| リスク | 対応 |
|---|---|
| 敵 300 体で 60fps を割る | 早期にストレステストシーンを作り、プール+空間ハッシュの実効性能を計測してから機能を積む |
| 迅移 Lv3(15.6 倍速)で衝突すり抜け | 迅移中はレイキャスト式の掃引判定に切り替える(04 で詳細化) |
| バランス調整の手戻り | データ駆動を徹底し、数値は全て `src/data/` に集約。デバッグ用の時間スキップ・無敵チートを最初から実装 |
